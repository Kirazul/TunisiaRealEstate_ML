01. Why did we use three datasets instead of only one?

Because one dataset gives volume, but the others add missing places and different listing patterns. If we use only one source, coverage becomes weaker in some areas. By combining the three cleaned sources, we keep more valid rows and a better national spread.

02. Why did we spend so much effort on cleaning first?

Because the raw listings were not ready for modeling. They had bad surfaces, missing fields, duplicates, mixed naming styles, and some unrealistic prices. If we skip cleaning, the merge and alignment stages become noisy and the model learns wrong patterns.

03. Why did we merge the datasets instead of training them separately?

The goal is one national pipeline, not three disconnected models. Merging lets us train one consistent model and build one atlas. We also remove cross-source duplicates during merge, so we keep the useful signal without counting the same listing many times.

04. Why was geographic alignment a major step in this project?

Because location is one of the strongest signals in real estate pricing. The raw files use many different names for the same place, so we needed a canonical geography layer. Without alignment, the model and atlas would treat the same area as many unrelated places.

05. Why did we add so many locality aliases and manual mappings?

Because many unmatched rows were not truly unknown places. They were common names written in a way that the base matcher could not connect directly, like `Ariana Ville`, `La Soukra`, `Le Bardo`, or `Sahloul`. The alias layer fixes these repeated real-world naming patterns.

06. Why was repairing geographic names so important for the dataset?

Because many rows were not truly bad rows. They were valid listings with noisy place names. Once those names were repaired with aliases and manual mappings, a large number of rows became usable again for both training and geographic coverage.

07. Why do we compare training rows and modeling rows?

Because it shows how much data is lost in the final modeling filter. The training rows tell us how much usable data we prepared, and the modeling rows tell us how much remains after the final outlier trim. This is a good sanity check for data efficiency.

08. Why did we keep longitude and latitude in the features?

Because price is spatial. Two properties with the same size can still have very different prices if they are in different places. Coordinates help the model capture location structure more smoothly than using only category labels.

09. Why do we still trim outliers in stage 8 after cleaning?

Cleaning removes obvious bad data, but some extreme prices still remain. Stage 8 trims the lowest and highest tails so the final model is not pulled by unusual rows. This makes the final price behavior more stable.

10. Why did we keep a relatively small feature set?

Because we wanted features that are strong, available, and easy to reproduce in production. It is better to keep a compact set of reliable features than to add many weak or unstable ones that make deployment harder.

11. Why did GradientBoosting win instead of RandomForest?

Because on this processed dataset it gave the better validation result. The model tournament is not based on opinion. Both models are trained on the same final dataset, then the pipeline keeps the one that performs better.

12. Why did we use cross-validation if we already have a test split?

The test split gives one final check, but cross-validation shows whether the model is stable across different folds. That is useful because one lucky split can look strong even if the model is not consistently reliable.

13. Why is atlas coverage 100% if direct support is only around 39%?

Because coverage and direct support mean different things. Coverage means the system can return a profile for the delegation. Direct support means the delegation has enough local examples to build that profile directly. The rest are handled through fallback logic.

14. What exactly is direct support in this project?

Direct support means a delegation has enough rows in the support pool for at least one property family, so its profile can be built from local delegation data. It is the strongest support level because it uses the same area instead of borrowing from a wider region.

15. How does fallback work exactly?

The fallback works in three levels. First, the system tries direct delegation support for the selected property family. If that delegation does not have enough local rows, it falls back to the governorate-level profile for the same family. If the governorate level is still not available, it falls back to the national profile for that family. This means the atlas can still answer everywhere, but it also keeps track of whether the answer came from exact local support, governorate fallback, or national fallback.

16. Why did we use target encoding for governorate and delegation?

Because location is very important, but using only one-hot encoding for every place is not enough. Target encoding gives the model a compact numeric summary of local price behavior. It helps the model learn price differences between areas more directly.

17. Why did we add the local price ratio feature?

Because it tells the model whether a property is cheaper or more expensive than the usual local level. This is useful because price per square meter can change a lot by area. The ratio gives a normalized local market signal instead of only raw price values.

18. Why did we keep both governorate and delegation in the model?

Because they capture different geographic scales. Delegation gives a more local signal, while governorate gives a broader regional signal. Keeping both helps the model stay useful even when local information is weaker in some places.

19. What are the main limits of the current pipeline?

The biggest limit is still the data source. These are listing prices, not official confirmed transactions. Also, a few locality names still need manual aliasing, and some geographic fallback is still necessary in weak areas.

20. What would improve the project the most next?

The best next improvement would be better verified transaction data. After that, the next strong improvement would be expanding the locality reference layer and adding more structural property features. That would improve both geographic confidence and model quality.
